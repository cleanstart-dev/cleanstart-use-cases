PLACEHOLDER — this file will be populated by scripts/03-scan-both.sh

The actual scan output goes here after you run:

    bash scripts/03-scan-both.sh

Expected format is the raw stdout of:

    oscap-docker image stig-retrofit:latest xccdf eval \
        --profile xccdf_org.ssgproject.content_profile_stig \
        --results results/retrofit-scan.xml \
        --report  results/retrofit-scan.html \
        /usr/share/xml/scap/ssg/content/ssg-ubuntu2204-ds.xml

The scan_image() function in 03-scan-both.sh redirects that stdout here.

============================================================
What you should expect to see when you read this file:
============================================================

For each rule, two lines like:

    Rule    xccdf_org.ssgproject.content_rule_<name>
    Result  pass | fail | notapplicable

At the bottom, a summary section with total counts.

============================================================
What to verify before publishing the comparison:
============================================================

1. The host running the scan has ssg-ubuntu2204-ds.xml installed
   and reachable at the path used in 03-scan-both.sh.

2. The SSG version on the host. Print with:
       oscap info /usr/share/xml/scap/ssg/content/ssg-ubuntu2204-ds.xml
   Record the version (e.g., 0.1.74) somewhere in the published post.

3. The Ubuntu 22.04 STIG benchmark release you are aligning to.
   The most recent DISA-published release at time of writing is V2R6
   (2026-01-28). SSG content tracks STIG releases but may lag by a
   release or two.

4. Several "fails" in any container scan are host-inherited controls
   (GRUB, partitioning, BIOS, kernel boot args). The DISA Container
   Hardening Process Guide says explicitly: "If an OpenSCAP scan
   returns noncompliant result(s), always evaluate the validity of
   those findings." Filter these before publishing pass/fail counts,
   or report them separately as "not applicable to containers".
