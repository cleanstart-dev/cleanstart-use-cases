# STIG Hardening: By Design vs. Retrofit

Same STIG mandate, two paths. One costs 354 lines of bash you own forever. The other costs a one-line `FROM`.

---

## Results (actual, reproduced locally)

| Metric | Retrofit (Ubuntu 22.04 + remediation) | By design (CleanStart) |
|---|---|---|
| Image size | 145 MB | 44.9 MB |
| Packages | 126 | none (no package manager) |
| Remediation script | 354 lines — your team owns it | none — vendor-maintained |
| Non-root default | yes (uid 65532) | yes (uid 1000) |
| Shell present | yes (`/bin/bash`) | no |
| Build time | ~83s | ~2s |

---

## Reproduce it

```bash
git clone https://github.com/cleanstart-dev/cleanstart-use-cases.git
cd cleanstart-use-cases/stig-by-design-vs-retrofit-v3/stig-by-design-vs-retrofit

bash scripts/run.sh
```

That's it. One script — builds both images, runs the comparison, writes logs to `results/`.

## The structural argument

The retrofit Dockerfile runs a 354-line bash script that rewrites PAM, sshd, auditd, sysctl, and modprobe configs. Every STIG benchmark update (quarterly), every new CVE in any included package, every engineer who leaves — that's your team's problem.

The by-design Dockerfile:

```dockerfile
FROM cleanstart/glibc:latest
COPY --chown=1000:1000 app /app
USER 1000
ENTRYPOINT ["/app/server"]
```

No shell. No package manager. No remediation script. 62 hardened images at [hub.docker.com/u/cleanstart](https://hub.docker.com/u/cleanstart).

---

## When retrofit is still the right call

- Legacy stack that genuinely needs packages a minimal base doesn't ship
- Air-gapped environments with no path to an external hardened base
- Single-tenant deployments needing exact host OS parity

For those cases, `scripts/retrofit-remediation.sh` is a working starting point.
