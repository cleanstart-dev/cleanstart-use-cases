# STIG Hardening: By Design vs. Retrofit

Same STIG mandate, two paths. One costs 354 lines of bash you own forever. The other costs a one-line `FROM`.

This repo builds both, scans both, and prints the difference. Every number below was produced by running `bash scripts/run.sh` — nothing is made up.

---

## What is STIG and why does it matter for containers?

**STIG** (Security Technical Implementation Guide) is a set of security configuration standards published by DISA (Defense Information Systems Agency) — the US Department of Defense's IT security body. Every federal agency, defense contractor, and a large portion of regulated industries (finance, healthcare, critical infrastructure) is required to meet STIG compliance for any system that touches sensitive data or government networks.

For containers specifically, DISA does not publish a standalone container STIG. Instead, container security is governed by the **DISA Container Hardening Process Guide**, which maps container requirements to the General Purpose Operating System SRG. In practice this means: if your container runs on Ubuntu, it must satisfy the Ubuntu 22.04 STIG — currently at **V2R6 with 300+ individual rules**.

### What does a STIG rule actually look like?

Each rule covers one specific security control. Examples from the Ubuntu 22.04 STIG:

- Passwords must use SHA-512 hashing with at least 5000 rounds
- SSH must be configured with `PermitRootLogin no` and `PermitEmptyPasswords no`
- The audit daemon must log all privilege escalation events
- Unused filesystems (cramfs, jffs2, hfs) must be disabled via modprobe blacklist
- Setuid binaries must be minimized
- Insecure legacy tools (telnet, ftp, rsh, talk, finger) must be removed

There are 300+ of these rules, and a new benchmark version ships roughly every quarter.

### The container hardening problem

The challenge is that general-purpose base images like `ubuntu:22.04` were built for flexibility, not compliance. They ship with shells, package managers, legacy tools, and permissive defaults — all of which STIG flags. Teams have two options:

1. **Retrofit** — take the general-purpose base and apply remediation on top (patch configs, remove packages, rewrite PAM/sshd/auditd settings)
2. **By design** — start from a base that was purpose-built to satisfy these controls from day one

This repo shows exactly what each path costs.

## Reproduce it

**Prerequisites:** Docker. Nothing else.

```bash
git clone https://github.com/cleanstart-dev/cleanstart-use-cases.git
cd cleanstart-use-cases/stig-by-design-vs-retrofit-v3/stig-by-design-vs-retrofit

bash scripts/run.sh
```

`run.sh` does everything in one shot:
1. Builds `stig-retrofit:latest` from `configs/Dockerfile.retrofit`
2. Builds `stig-by-design:latest` from `configs/Dockerfile.by-design`
3. Inspects both images — package count, user, shell
4. Prints the side-by-side comparison table
5. Writes build logs to `results/`

## The two paths explained

### Path 1 — Retrofit

Start from `ubuntu:22.04` and apply STIG hardening on top. This is what most teams do when handed a STIG mandate against an existing image.

`configs/Dockerfile.retrofit` does this in 5 steps:
1. Remove packages flagged by STIG (telnet, ftp, rsh, talk, finger, NIS)
2. Install what STIG requires — `auditd`, `libpam-pwquality`, `aide`
3. Run `scripts/retrofit-remediation.sh` — 354 lines of config rewrites across PAM, sshd, auditd, sysctl, modprobe, AIDE, banners, cron
4. Add a non-root user (Ubuntu ships as root by default)
5. Tighten file permissions on shadow, passwd, sshd_config

The result is a 145 MB image with 126 packages, a bash shell, and a remediation script that must be updated every time STIG releases a new benchmark version (roughly quarterly).

### Path 2 — By Design

Start from a base that was already built to satisfy STIG-aligned controls as architectural properties.

`configs/Dockerfile.by-design` is 4 lines:

```dockerfile
FROM cleanstart/glibc:latest
COPY --chown=1000:1000 app /app
USER 1000
ENTRYPOINT ["/app/server"]
```

The result is a 44.9 MB image with no package manager, no shell, and a non-root user — out of the box, no remediation script needed.

## What the retrofit path cannot fix

Even after running all 354 lines of remediation, the retrofit image still has structural findings that cannot be removed without breaking the image:

| Package / binary | Why it cannot be removed |
|---|---|
| `wget`, `curl`, `tar` | Required during image build or runtime operations |
| `dpkg`, `apt` | Removing breaks OS package management the image depends on |
| `/usr/bin/bash` | Wired into core Ubuntu utilities |
| ~20 setuid binaries (`mount`, `su`, `sudo`, `ping`) | Pruning these breaks the image |
| `perl`, `python3` | Pulled in transitively by apt and admin tooling |

The by-design image ships without all of these — which is why it is 69% smaller and has no shell.

## The lifecycle cost

A point-in-time scan flatters the retrofit path. The real cost shows up over 12 months:

| Event | Retrofit | By design |
|---|---|---|
| New CVE in `wget` (still present) | Rebuild, re-test, re-scan, re-deploy | Not in image — no action needed |
| STIG benchmark version bump | Re-read diff, update script, re-test | Vendor pushes base update; you rebuild |
| Engineer who owned the script leaves | Tribal knowledge loss | Vendor maintains baseline |
| New team needs a STIG image | Fork and maintain another copy of the script | Pull the base, ship the app |

---

## When retrofit is still the right call

This is not "by-design always wins":

- **Legacy stack** that genuinely needs packages a minimal base doesn't ship
- **Air-gapped environments** with no path to consume an external hardened base
- **Single-tenant deployments** needing exact parity with a host OS image

For those cases, `scripts/retrofit-remediation.sh` is a working starting point — fork it, harden further, own the maintenance.
